# Contributing to SuperPower with Files (SPF) 🚀

First off, thank you for considering contributing to SPF! It's people like you that make SPF such a powerful tool for the community.

## 🏗 Repository Architecture

SPF is a **unified workflow** built on top of modular components. Understanding how we manage code is key to contributing:

- **`/vendor`**: Contains raw upstream submodules. **Do not edit these directly.**
- **`/overlays`**: This is where the magic happens. We use markdown overlays to inject SPF-specific rules (like memory tracking and branding) into upstream skills.
- **`/skills`**: The **built** output. This directory is generated by the build script and should not be edited manually.
- **`/src/hooks`**: Custom project-specific logic for session initialization and completion checks.
- **`/scripts`**: Utility scripts, including our core build engine.

## 🛠 Development Workflow

### 1. Requirements
- Git
- Bash-compatible shell

### 2. Making Changes
If you want to modify a skill's behavior:
1.  **Identify the Target**: Find the skill in `/vendor` or `/skills`.
2.  **Edit/Create Overlay**: Add your changes to a relevant `.md` file in `/overlays/superpowers/`. Use the `<!-- target: ... -->` and `<!-- action: append/overwrite -->` syntax at the top of the file.
3.  **Build**: Run the build script to merge your overlays into the final skills:
    ```bash
    ./scripts/build_unified.sh
    ```
4.  **Verify**: Check the generated file in `/skills/` to ensure your overlay applied correctly.

### 3. Submission Process
1.  **Fork** the repository.
2.  **Create a feature branch** (`git checkout -b feature/amazing-new-skill`).
3.  **Commit your changes** (`git commit -m 'feat: add support for X'`).
4.  **Push to the branch** (`git push origin feature/amazing-new-skill`).
5.  **Open a Pull Request**.

## 🖋 Coding Standards
- **SPF Branding**: Always use the full name `.superpower-with-files/` for default directories.
- **Announcements**: Skills must announce themselves using `🚀 **SUPERPOWER ACTIVE:** skill-name`.
- **Memory Tracking**: All process-heavy skills should involve updating `task_plan.md` or `progress.md`.
- **Timestamps**: Append UTC timestamps to memory files after modifications.

## ⚖️ Path Precedence Rule
All skills must honor this priority for file operations:
1.  **User Request [HIGHEST]**: If the user specifies a path, use it.
2.  **SPF Default [SECONDARY]**: Use `.superpower-with-files/`.
3.  **Legacy Paths [LOWEST]**: Avoid using hardcoded paths like `docs/plans/` unless explicitly requested.
